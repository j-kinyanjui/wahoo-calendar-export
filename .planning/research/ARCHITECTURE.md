### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| `Application.module()` | Entry point, wires plugins | Ktor Application module |
| `RestClient.kt` | Shared HTTP client for REST calls | Ktor HttpClient (CIO) |
| `WahooAuthenticationOauth2.kt` | OAuth2 flow, session management | Ktor Auth plugin + Sessions |
| `GraphQLKtorClient` | GraphQL queries to Systm API | GraphQL Kotlin library |
| `SystmGraphQLClient.kt` | Systm-specific GraphQL setup | Custom wrapper with JWT auth |
| `SystmAuthentication.kt` | JWT token storage/retrieval | Ktor Sessions or dedicated config |
| `Models` | Data models | kotlinx.serialization |

### Structure Rationale

- **`plugins/`: Ktor plugin idiom** — aligns with existing pattern for organizing server setup
- **`graphql/queries/`: Convention for GraphQL Kotlin** — plugin looks for .graphql files here
- **`graphql/generated/`: Build output** — auto-generated by GraphQL Kotlin Gradle plugin
- **`routes/`: Separation of concerns** — routing logic separate from authentication

## Architectural Patterns

### Pattern 1: Shared HttpClient

**What:** Reuse a single Ktor HttpClient instance across REST and GraphQL clients
**When:** Multiple HTTP-based integrations exist in same application
**Trade-offs:** PRO: Resource efficiency, shared connection pooling. CON: Single client config affects all integrations.

```kotlin
// RestClient.kt (existing)
val applicationHttpClient = HttpClient(CIO) {
    install(ContentNegotiation) { json() }
}

// SystmClient.kt (new)
val systmHttpClient = HttpClient(CIO) {
    engine { requestTimeout = 60_000 }
    install(ContentNegotiation) { json() }
}
val graphQLClient = GraphQLKtorClient(
    url = URL("https://api.thesufferfest.com/graphql"),
    httpClient = systmHttpClient
)
```

### Pattern 2: Per-Request Authentication

**What:** Inject auth headers per-request rather than globally
**When:** Different endpoints require different auth mechanisms (Wahoo OAuth2 vs Systm JWT)
**Trade-offs:** PRO: Clear separation, flexible. CON: More boilerplate per call.

```kotlin
// Per-request Bearer token
val query = GetUserPlansRange(variables = GetUserPlansRange.Variables(...))
val result = graphQLClient.execute(query) {
    header(HttpHeaders.Authorization, "Bearer $jwtToken")
}
```

### Pattern 3: Plugin-Based Configuration

**What:** Encapsulate related functionality in Ktor plugins
**When:** Organizing related routing, authentication, and configuration
**Trade-offs:** PRO: Clean separation, testable. CON: Slight indirection.

```kotlin
fun Application.configureSystm(httpClient: HttpClient = applicationHttpClient) {
    val graphQLClient = GraphQLKtorClient(
        url = URL("https://api.thesufferfest.com/graphql"),
        httpClient = httpClient
    )
    routing {
        get("/systm/plans") {
            token val = call.sessions.get<SystmSession>()?.token
            val plans = getSystmPlans(graphQLClient, token)
            call.respond(plans)
        }
    }
}
```

## Data Flow

### Request Flow: Systm GraphQL

```
[User requests /systm/plans]
        ↓
[SystmRoutes] → Validates session, retrieves JWT
        ↓
[GraphQLKtorClient] → Builds request with Bearer token
        ↓
[HTTP POST api.thesufferfest.com/graphql] → GraphQL query
        ↓
[Response] → kotlinx.serialization deserializes
        ↓
[Transform to internal model] → Extract workout names/dates
        ↓
[Response to user]
```

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 0-1k users | Single HttpClient instances, on-demand sync — monolith is fine |
| 1k-100k users | Connection pooling tuning, consider caching GraphQL responses |
| 100k+ users | Rate limiting per user, separate GraphQL client pools, async job queue for sync |

### Scaling Priorities

1. **First bottleneck:** API rate limits from Systm — cache training plans locally, sync on schedule
2. **Second bottleneck:** JWT token management — consider token refresh mechanism if supported

## Anti-Patterns

### Anti-Pattern 1: Creating New HttpClient Per Request

**What people do:** `HttpClient() { ... }.get(...)` inside each route
**Why it's wrong:** Connection overhead, resource exhaustion under load
**Do this instead:** Reuse shared HttpClient instance

### Anti-Pattern 2: Hardcoding GraphQL Endpoint

**What people do:** `URL("https://api.thesufferfest.com/graphql")` inline
**Why it's wrong:** No environment flexibility, harder to test
**Do this instead:** Load from config: `environment.config.property("systm.graphqlUrl").getString()`

### Anti-Pattern 3: Mixing Auth Mechanisms

**What people do:** Using OAuth2 headers for GraphQL or vice versa
**Why it's wrong:** Wrong auth = 401 errors, confusing debugging
**Do this instead:** Explicit per-request headers or dedicated client instances per auth type

## Build Order (Suggested)

1. **Add dependencies** — `graphql-kotlin-ktor-client` + GraphQL Gradle plugin
2. **Create GraphQL query file** — `GetUserPlansRange.graphql` in `src/main/resources`
3. **Configure Gradle plugin** — Point to Systm schema, generate types
4. **Create SystmClient plugin** — GraphQLKtorClient wrapper with JWT auth
5. **Create SystmRoutes** — Endpoint to fetch plans via GraphQL
6. **Create authentication** — JWT token storage (session or config)
